Project 1

A Chat Application for Remote Message Exchange

A simple chat application that utilizes socket programming.

Written in C and compiles in a Linux system environment.

Uses TCP Sockets in the peer connection implementation and uses the select() API for handling multiple socket connections.

I integrated both client-side and server side code into one program and ran on each peer.

The process takes one command line parameters. The parameter indicates the port on which your process will listen for the incoming connections. For example, you can run it like this: ./chat 4322, where 4322 is the listening port.

When launched, the process works like a UNIX shell. It accepts incoming connections and at the same time provides a user interface that will offer the following command options:

  1. help- Display information about the available user interface options or command manual.
  2. myip- Display the IP address of this process.
  3. myport- Display the port on which this process is listening for incoming connections.
  4. connect <destination> <port no>>- This command establishes a new TCP connection to the specified <destination> at the specified <port no>. The <destination> is the IP address of the computer. Any attempt to connect to an invalid IP is rejected and suitable error messages are displayed.
  5. list>- Display a numbered list of all the connections this process is part of. This numbered list will include connections initiated by this process and connections initiated by other processes. The output should display the IP address and the listening port of all the peers the process is connected to.
  6. terminate <connection id.>>- This command will terminate the connection listed under the specified number when LIST is used to display all connections.
  7. send <connection id.> <message>>- (For example, send 3 Oh! This project is a piece of cake). This will send the message to the host on the connection that is designated by the number 3 when command “list” is used. The message to be sent can be up-to 100 characters long, including blank spaces. On successfully executing the command, the sender displays “Message sent to <connection id.>” on the screen.
  8. exit>- Close all connections and terminate this process. The other peers also update their connection list by removing the peer that exits.

Project 2

QuikQuisine- Android Application

Currently Quik Quisine is an app that allows people who have limited ingredients to find recipes based on the ingredients, as well as buy missing ingredients. It allows users to specify what types of recipes they would like to make with the ingredients as well such as vegetarian, desserts, etc. It also allows them to rate and review different recipes.

At the time this app was designed the world was in a pandemic. We decided to design an app that could help people who were perhaps low on income and needed to scramble a meal together with whatever they had at home.

Quik Quisine is the solution to these problems as it would show users how to create recipes with the few ingredients they have in the house. Not only that but it would also allow users to be able to order missing ingredients through the app. With both issues dealt with it would allow more people to save money by being able to cook at home.

Designed with Flutter and Dart to develop cross platform applications for Android and iOS mobile devices and utilizes Rails API and MySQL to set up the server.

Functionality includes:

  • the creation of user profiles
  • recommending recipes based on user's taste and trends
  • the ability to order ingredients to be delivered to the user's home throught the use of third party APIs
  • the creation of roles to allow chefs to create recipes
  • the ability to follow famous chefs and save recipes
  • a recipe review system
  • a the ability to search and filter the database for recipes

Project 3

Distance Vector Routing Protocol

This is a simplified version of the Distance Vector Routing Protocol that runs on 4 four servers/laptops (behaving as routers) using TCP.

The servers are able to output their forwarding tables along with the cost and be robust to link changes. A server sends out routing packets in the following two conditions: a) periodic update and b) the user uses command asking for one.

In this environment, costs are bi-directional. Whenever a new server is added to the network, it will read its local topology file to determine who are its neighbors. Routing updates are exchanged periodically between neighboring servers. When this newly added server sends routing messages to its neighbors, they will add an entry in their routing tables corresponding to it. When a server has been removed from a network, it will no longer send distance vector updates to its neighbors.

Use the following command options:

  1. server -t <topology-file-name> -i <routing-update-interval>>- topology-file-name: The topology file contains the initial topology configuration for the server. routing-update-interval: It specifies the time interval between routing updates in seconds. port and server-id: They are written in the topology file. The server should find its port and server-id in the topology file without changing the entry format or adding any new entries.
  2. update <server-ID1> <server-ID2> <Link Cost>>- server-ID1, server-ID2: The link for which the cost is being updated. Link Cost: It specifies the new link cost between the source and the destination server. Note that this command will be issued to both server-ID1 and server-ID2 and involve them to update the cost and no other server.
  3. step>- Send routing update to neighbors right away. Note that except this, routing updates only happen periodically.
  4. disable <sever-ID>>- Disable the link to a given server. Doing this “closes” the connection to a given server with server-ID.
  5. crash>- Close all connections. This is to simulate server crashes. Close all connections on all links. The neighboring servers set the link cost to infinity.

Project 4

Graph with weights

A Java program that allowed users to create graphs by inputing nodes and weights.

Once the user has finished inputing their edge data, the program will apply several algorithms for traversing data structures.

The three algorithms are Depth First Search, Breadth First Search, and Minnimum Spanning Tree.

The program will illustrate the traversal order to reach every node in the graph.
